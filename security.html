Introduction

At Thehackitect Solutions Limited ("TSL"), security is fundamental to everything we do. This Security Policy outlines our comprehensive approach to protecting the confidentiality, integrity, and availability of our systems, infrastructure, and the sensitive data entrusted to us by our clients and stakeholders.

As a Nigerian technology company registered under RC 7234567, we are committed to maintaining the highest security standards in compliance with the Nigeria Data Protection Regulation (NDPR), international best practices, and industry-recognized security frameworks.

This policy applies to all TSL employees, contractors, partners, systems, networks, and services. We continuously evolve our security practices to address emerging threats and maintain robust protection for our clients' digital assets.

1. Information Security Framework

Our security program is built on industry-leading frameworks and standards:

ISO 27001 Principles

Information security management system best practices

NDPR Compliance

Full adherence to Nigerian data protection regulations

OWASP Guidelines

Web application security top 10 mitigation strategies

NIST Framework

Cybersecurity framework for risk management

Our security framework encompasses five core pillars: Identify, Protect, Detect, Respond, and Recover. We implement defense-in-depth strategies with multiple layers of security controls to safeguard against sophisticated threats.

2. Data Security Measures

2.1 Data Classification

We classify all data into four categories based on sensitivity:

  • Critical: Payment information, authentication credentials, personal identification numbers (highest protection)
  • Confidential: Client contracts, proprietary software code, business strategies
  • Internal: Employee information, internal communications, operational data
  • Public: Marketing materials, published content, public website information

2.2 Data Storage Security

Encrypted Storage

All data at rest is encrypted using AES-256 encryption standard

Secure Infrastructure

Data hosted in secure, tier-III certified data centers with redundancy and backup systems

Database Security

Database encryption, parameterized queries, and SQL injection protection

Automated Backups

Daily encrypted backups with 30-day retention and off-site storage

2.3 Data Retention and Disposal

We retain data only as long as necessary for business and legal purposes. Obsolete data is securely disposed using cryptographic erasure for digital data and certified shredding for physical media. Disposal activities are logged and auditable.

3. Encryption Practices

3.1 Data in Transit

TLS 1.3 Encryption

All data transmitted over networks uses TLS 1.3 (Transport Layer Security) with strong cipher suites. We enforce HTTPS for all web communications.

Certificate Pinning

Mobile and desktop applications implement certificate pinning to prevent man-in-the-middle attacks.

VPN Security

Remote access to internal systems requires VPN connections with multi-factor authentication.

3.2 Data at Rest

AES-256 Encryption

All stored data is encrypted using Advanced Encryption Standard (AES) with 256-bit keys.

Field-Level Encryption

Highly sensitive fields (passwords, payment details) receive additional field-level encryption.

Key Management

Encryption keys are managed using Hardware Security Modules (HSMs) and rotated regularly according to best practices.

3.3 Password Security

User passwords are hashed using bcrypt or Argon2 with per-user salts. We never store passwords in plaintext or reversible encryption. Password policies enforce minimum complexity requirements: minimum 12 characters, mix of uppercase, lowercase, numbers, and special characters.

4. Access Controls and Authentication

4.1 Principle of Least Privilege

Access rights are granted based on the minimum level required to perform job functions. Users receive only the permissions necessary for their role, reducing potential attack surfaces. Access is reviewed quarterly and immediately revoked upon role changes or termination.

4.2 Multi-Factor Authentication (MFA)

MFA is mandatory for:

  • All administrative and privileged accounts
  • Access to production systems and databases
  • Remote access to corporate networks
  • Client portals and sensitive applications
  • Financial systems and payment processing interfaces

We support multiple MFA methods including TOTP authenticator apps, SMS codes, hardware tokens, and biometric authentication.

4.3 Identity and Access Management (IAM)

Single Sign-On (SSO): Centralized authentication for internal applications reducing password fatigue

Role-Based Access Control (RBAC): Permissions assigned based on organizational roles and responsibilities

Access Logging: All access attempts, successful and failed, are logged and monitored

Session Management: Automatic session timeouts, secure session tokens, and session invalidation controls

4.4 Physical Security

Our offices and data centers employ physical security controls including: biometric access systems, 24/7 security personnel, CCTV surveillance, visitor logging and escort policies, secure equipment disposal procedures, and restricted access zones for sensitive operations.

5. Network Security

Firewalls

Next-generation firewalls with intrusion prevention, deep packet inspection, and application-layer filtering

IDS/IPS

Intrusion Detection and Prevention Systems for real-time threat detection and automatic blocking

DDoS Protection

Cloud-based DDoS mitigation services protecting against volumetric and application-layer attacks

Network Segmentation

Isolated network zones separating production, development, and administrative networks

Vulnerability Scanning

Automated weekly network vulnerability scans identify misconfigurations, missing patches, and security weaknesses. Critical vulnerabilities are remediated within 24 hours.

Wireless Security

Enterprise Wi-Fi networks use WPA3 encryption, 802.1X authentication, network isolation for guest access, and regular rogue access point detection.

6. Application Security

Secure Development Lifecycle (SDL)

Security is integrated throughout our development process:

Requirements Analysis Threat Modeling Secure Coding Code Review Security Testing Penetration Testing Deployment Security

Code Security Measures

  • Static Application Security Testing (SAST) integrated into CI/CD pipelines
  • Dynamic Application Security Testing (DAST) for runtime vulnerability detection
  • Software Composition Analysis (SCA) for third-party dependency vulnerabilities
  • Mandatory peer code reviews before merging to production branches
  • Input validation, output encoding, and parameterized queries to prevent injection attacks
  • Content Security Policy (CSP) headers and CORS configuration

API Security

All APIs implement OAuth 2.0/JWT authentication, rate limiting to prevent abuse, request validation and sanitization, API versioning for backward compatibility, and comprehensive logging of API access and usage patterns.

We subscribe to global threat intelligence feeds, participate in information sharing communities, and maintain awareness of emerging threats. Our defenses are continuously updated based on the latest threat landscape.

8. Incident Response Procedures

8.1 Incident Response Plan

Our formal incident response plan follows industry best practices:

1

Identification

Detect and confirm security incidents through monitoring, alerts, or reports

2

Containment

Isolate affected systems to prevent spread and limit damage

3

Eradication

Remove threat actors, malware, and vulnerabilities from environment

4

Recovery

Restore systems and services to normal operations with enhanced security

5

Lessons Learned

Post-incident review to improve security posture and prevent recurrence

8.2 Incident Communication

In the event of a security incident affecting client data, we commit to notifying affected parties within 72 hours as required by NDPR. Communications include incident description, data potentially impacted, mitigation measures taken, and recommendations for affected users.

8.3 Business Continuity and Disaster Recovery

We maintain comprehensive business continuity plans with Recovery Time Objectives (RTO) of 4 hours and Recovery Point Objectives (RPO) of 1 hour for critical systems. Regular disaster recovery drills ensure plan effectiveness. Geographically distributed backups ensure data availability even in catastrophic scenarios.

11. Responsible Vulnerability Disclosure Program

11.1 Security Researcher Program

We welcome and encourage responsible disclosure of security vulnerabilities. Security researchers who discover potential issues are invited to report them confidentially to our security team.

How to Report a Vulnerability

Email: security@thehackitectsolutions.com

PGP Key: Available upon request for encrypted communications

Response Time: Initial acknowledgment within 48 hours

11.2 What to Include in Your Report

  • Detailed description of the vulnerability
  • Steps to reproduce the issue
  • Potential impact and severity assessment
  • Proof-of-concept (if applicable and safe)
  • Any suggested mitigation or remediation
  • Your contact information for follow-up

11.3 Our Commitments

Acknowledge receipt of your report within 48 hours

Provide regular updates on remediation progress

Credit researchers in our security advisories (with permission)

Not pursue legal action against researchers acting in good faith

Consider bug bounty rewards for qualifying vulnerabilities

11.4 Responsible Disclosure Guidelines

We request that security researchers:

  • Report vulnerabilities privately before public disclosure
  • Allow reasonable time for remediation (typically 90 days)
  • Avoid accessing, modifying, or deleting user data
  • Avoid service disruption or degradation
  • Do not exploit vulnerabilities beyond proof-of-concept
  • Respect user privacy and data confidentiality
